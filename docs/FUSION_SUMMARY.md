# OpenClaw 融合项目总结

> **项目完成时间**: 2026-04-27
> **项目分支**: feature/fusion-claude-hermes
> **核心理念**: 强化 OpenClaw，使其具备强大的自主持久工作能力和自主技能形成能力

---

## 一、项目背景与目标

### 1.1 融合动机

OpenClaw 是一个功能完整的 AI Agent 平台，但在实际使用中发现两个关键能力缺口：

| 缺口                 | 表现                                                       | 影响                 |
| -------------------- | ---------------------------------------------------------- | -------------------- |
| **自主持久工作能力** | Agent 无法长时间稳定完成复杂编程任务，容易中断或陷入死循环 | 复杂任务完成率低     |
| **自主技能形成能力** | Agent 无法从有效工作流程中"学习"并固化可复用技能           | 重复探索，效率不提升 |

### 1.2 融合来源

我们研究了两个优秀项目来解决这些问题：

**Claude Code** (Anthropic 官方 CLI):

- **12层 Harness 机制**: 让 Agent 能长时间稳定完成复杂编程任务
- 核心创新：StreamingToolExecutor、Fork 缓存优化、Coordinator 空闲循环、Background Tasks

**Hermes Agent** (开源 Agent):

- **skill_manage 工具**: Agent 从有效流程中学习并固化可复用技能
- 核心创新：学习触发检测、工作流程提取、规则持久化

### 1.3 项目目标

```
目标: 强化 OpenClaw
├── 自主持久工作能力 (来自 Claude Code)
│   ├── 权限安全决策 (Phase 6)
│   ├── 上下文管理 (Phase 2)
│   ├── 记忆系统 (Phase 1)
│   ├── 终端安全执行 (Phase 3)
│   ├── 流式并发执行 (Phase 10)
│   ├── Fork 缓存优化 (Phase 11)
│   ├── Coordinator 空闲循环 (Phase 12)
│   └── Background Tasks (Phase 13)
│
└── 自主技能形成能力 (来自 Hermes Agent)
    ├── skill_manage 工具 (Phase 8)
    └── 规则持久化 (Phase 9)
```

## 三、与原版 OpenClaw 的区别

### 3.1 新增能力

| 能力                 | 原版 OpenClaw   | 融合版 OpenClaw               | 增益             |
| -------------------- | --------------- | ----------------------------- | ---------------- |
| **记忆系统**         | 无结构化记忆    | 类型化记忆 + 自动索引         | Agent 可积累知识 |
| **熔断器**           | 无失败保护      | 连续失败自动熔断              | 防止死循环       |
| **危险命令检测**     | 无安全检测      | 30+ 模式检测 + 审批流程       | 安全执行         |
| **权限审批**         | 简单 allow/deny | PermissionMode + Gateway 审批 | 用户可控安全     |
| **技能学习**         | 手动创建技能    | Agent 自主创建/编辑技能       | 持续进化         |
| **批量 Spawn**       | 单个 spawn      | 批量 spawn + Prompt Cache     | 效率提升         |
| **Coordinator**      | 无空闲循环      | 空闲自动认领任务              | 持久工作         |
| **Background Tasks** | 无后台执行      | 后台持久执行                  | 长时间任务       |

### 3.2 架构差异处理

Claude Code 的某些核心设计不适用于 OpenClaw 的架构，我们采用外围适配策略：

| 原始设计              | 原始位置            | OpenClaw 适配位置              | 原因                              |
| --------------------- | ------------------- | ------------------------------ | --------------------------------- |
| StreamingToolExecutor | 替换 Tool.ts 执行器 | concurrency-control-wrapper.ts | SessionManager 库控制执行         |
| Fork Cache            | 批量 Fork 消息构建  | sessions-spawn-batch-tool.ts   | OpenClaw 使用 session-based spawn |
| Coordinator 空闲循环  | run.ts 主循环       | coordinator-loop-manager.ts    | OpenClaw 运行完成即返回           |

**策略**: 不修改 OpenClaw 核心代码（run.ts、SessionManager），通过外围层实现相同能力。

### 3.3 sandbox 模块职责分离

原计划将 terminal 模块与 sandbox 模块统一，最终采用方案 C（职责分离）：

| 模块               | 职责                                          | 代码位置                   |
| ------------------ | --------------------------------------------- | -------------------------- |
| terminal           | 命令级安全检测 (30+ 模式) + local 执行        | `src/agents/terminal/`     |
| sandbox            | 容器级隔离 (Docker/SSH) + workspace lifecycle | `src/agents/sandbox/`      |
| bash-tools.exec.ts | 桥接协调：危险检测 → sandbox 执行             | Line 1511-1526 → Line 1688 |

**验证**: 危险检测在 sandbox 执行之前生效 ✅
